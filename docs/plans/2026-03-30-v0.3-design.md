**Goal:** Expand test coverage for validated disk images, fix ADV audio, improve SD card UX with subdirectory browsing, tune 6502 performance based on profiling, and reduce `app_main.c` complexity by extracting modules.

**Architecture:** Two checkpoints. Checkpoint 1 covers testing and hardware fixes (items 1–3). Checkpoint 2 covers UX, performance, and code health (items 4–6). Each checkpoint ends with a verification step.

## Checkpoint 1: Test Coverage & Compatibility

### 1. Smoke Test Entries for Validated Disks

**Problem:** The emulator boots many disk images correctly, but there are no automated regression tests for them. Only the DOS 3.3 system disk is covered by the default smoke test.

**Solution:** Add `tests/run_disk_compat.sh` — a new script that tests each validated disk image with its known-good parameters. Each test is skipped if the disk file is not present in `disks/`. This keeps the fast test path (`run_tests.sh` + `run_rom_smoke.sh`) unchanged.

Test matrix:

|-----------|-----------|-------------|-------|-------|
| `VisiCalc_1984_Software_Arts.do` | `DO YOU WANT TO USE 80 COLUMNS` | — | 50M | DOS 3.3 app |
| `Lemonade_Stand_1979_Apple.do` | `LEMONADE` | — | 50M | DOS 3.3 app |
| `choplifter.dsk` | `CHOPLIFTER` | — | 50M | DOS 3.3 app |
| `bards_tale_boot.dsk` | `DON'T BREAK` | ` ` (space) | 50M | Custom loader, needs keypress |
| `Oregon_Trail_Disk_1_of_2.dsk` | — | ` ` (space) | 200M | Custom loader, REQUIRE_NOT_BASIC_PROMPT |

Script structure:
- Source a shared helper that runs `run_rom_smoke.sh` with env vars and prints PASS/SKIP/FAIL per disk.
- Exit 0 if all present disks pass (skips are OK).
- Exit 1 if any present disk fails.

### 2. ADV Audio (ES8311 Codec)

**Problem:** The I2S audio pipeline works on the ADV (bytes written, ESP_OK), and ES8311 I2C register writes succeed, but no analog audio output is produced. Audio works on the Original Cardputer (NS4168, no codec needed).

**Root cause hypothesis:** The ES8311 codec configuration doesn't match what M5Unified uses. Multiple variables differ: I2S port number, sample rate, I2S data format, and init ordering. Previous attempts changed these one at a time but the I2C SDA/SCL pins were also swapped during some attempts, confounding results.

**Solution:** Isolate the ADV audio path cleanly from the Original:

1. **Variant-specific I2S config in `cardputer_audio_init`:**
   - Original: `I2S_NUM_0`, 22050 Hz, mono (confirmed working)
   - ADV: `I2S_NUM_1`, 48000 Hz, matching M5Unified

2. **ES8311 register sequence** (matching M5Unified `_speaker_enabled_cb_cardputer_adv`):
   - REG00=0x80 (power on CSM)
   - REG01=0xB5 (MCLK from BCLK, enable DAC clocks)
   - REG02=0x18 (PLL pre_div=1, pre_multi per M5Unified)
   - REG0D=0x01 (power up analog)
   - REG12=0x00 (power up DAC)
   - REG13=0x10 (enable HP driver)
   - REG32=0xBF (DAC volume 0 dB)
   - REG37=0x08 (bypass DAC EQ)

3. **Init ordering:** Enable I2S channel BEFORE writing ES8311 registers (BCLK must be running for PLL lock).

4. **I2C bus sharing:** The keyboard driver already initializes I2C_NUM_0 on SDA=8/SCL=9. Skip `i2c_param_config` and `i2c_driver_install` if the driver is already running (ESP_ERR_INVALID_STATE).

5. **If stereo is needed:** Try `I2S_SLOT_MODE_STEREO` as a fallback — the ES8311 may expect data on both left and right slots.

6. **Kconfig:** `CONFIG_M5APPLE2_AUDIO_SAMPLE_RATE` stays at 22050 in `sdkconfig.defaults`. `sdkconfig.defaults.adv` overrides to 48000.

### 3. ProDOS Smoke Test Support

**Problem:** The smoke test only validates DOS 3.3 boot patterns. ProDOS disks (AppleWorks .po) boot successfully but fail the stage1/stage2 memory checks.

**Solution:** Extend `apple2_rom_smoke.c` to handle ProDOS boot verification:

- For `.po` images with `APPLE2_TEST_EXPECT_TEXT` set: skip the DOS 3.3 stage1/stage2 checks entirely and rely on screen text matching (same as the custom-loader path).
- For `.po` images without `EXPECT_TEXT`: check for ProDOS boot progress by verifying the PC has left the boot ROM area and memory at $2000+ is populated (ProDOS kernel loads there). Skip the DOS 3.3 preload/stage2/prompt checks.
- Add AppleWorks 1.3 to the compat test matrix if a suitable EXPECT_TEXT can be identified (it may need two disks — boot + program — which the single-drive smoke test can't handle; in that case, skip it).

## Checkpoint 2: UX, Performance & Code Health

### 4. SD Subdirectory Browsing

**Problem:** The SD picker only shows files in the root directory. Users with many disk images organize them in folders.

**Solution:** Extend the picker to support directory navigation:

- **Directory entries in the picker list:** Directories appear sorted before files, prefixed with `/` (e.g., `/games`). The existing "builtin" and "empty" entries remain at the top.
- **Navigation:** `Enter` on a directory descends into it. `ESC` goes up one level. `ESC` at root exits the picker (existing behavior preserved).
- **Current path state:** Add a path stack to the picker state (array of directory name offsets into a path buffer, max depth 4). Track the current browse path as a null-terminated string (e.g., `/games/rpg`).
- **On-demand scanning:** Scan directory contents when entering it, replacing the current file list. Do not scan all directories at startup — only scan root at startup (preserving current behavior).
- **Depth limit:** 4 levels maximum to avoid stack overflow on ESP32. Deeper directories are not shown.
- **Auto-mount:** Unchanged — startup auto-mount only considers root-level images.
- **File list memory:** Reuse the existing `s_sd_disks` array. When entering a subdirectory, rescan and repopulate the array with that directory's contents. When leaving, rescan the parent. This avoids additional memory allocation.

### 5. 6502 Performance Tuning

**Problem:** The 6502 interpreter and bus read/write paths are the emulator's hot loop. The backlog identifies them as optimization targets, but no profiling data exists.

**Solution:** Profile-driven optimization in three steps:

**Step 1 — Benchmark harness:**
Add `tests/run_bench.sh` and a benchmark mode to `apple2_rom_smoke.c` (or a new `apple2_bench.c`). The benchmark:
- Loads the DOS 3.3 system disk and ROM
- Runs for a fixed number of instructions (e.g., 20M)
- Reports wall-clock time, instructions/sec, and emulated MHz
- Controlled by `APPLE2_TEST_BENCH=1` env var or a dedicated binary

**Step 2 — Profile:**
Use macOS Instruments (Time Profiler) or `perf` on the benchmark binary. Identify the top 5 hot functions. Expected candidates:
- `cpu6502_step` — opcode dispatch overhead
- `apple2_bus_read` / `apple2_bus_write` — per-access I/O range checks
- `apple2_disk2_tick` — called every instruction

**Step 3 — Targeted optimizations** (apply only what profiling justifies):
- **Fast-path RAM reads:** In `apple2_bus_read`, check `address < 0xC000` first and return `machine->memory[address]` directly. The I/O space check ($C000-$C0FF) is only needed for ~0.1% of reads.
- **Inline common addressing modes:** Flatten zero-page, immediate, and absolute addressing into the opcode switch arms instead of calling helper functions.
- **Reduce `disk2_tick` overhead:** Skip the tick entirely when the motor is off (already done) and when no disk is loaded (add early check on `loaded[drive]` before accumulator math).

Each optimization is validated by re-running the benchmark and confirming improvement.

### 6. app_main.c Refactoring

**Problem:** `app_main.c` is 3K+ LOC mixing emulation integration, SD card management, disk picker UI, and performance logging. This makes it hard to modify any one concern.

**Solution:** Extract three modules, each with a clear interface:

**`app_perf.c` / `app_perf.h`** (extract first — smallest, fewest dependencies):
- Owns the `s_perf` struct and all counter fields
- Exports `app_perf_reset()`, `app_perf_accumulate_*()`, `app_perf_log()`
- Main loop calls accumulate functions; periodic logging calls `app_perf_log()`

**`app_sd.c` / `app_sd.h`** (extract second):
- Owns `s_sd_disks[]`, `s_sd_disk_count`, `s_sd_drive_index[]`, `s_sd_drive_files[]`
- Exports `app_sd_init()`, `app_sd_scan()`, `app_sd_mount_disk()`, `app_sd_cycle_drive()`, `app_sd_close_drive()`, `app_sd_read_sector()`, `app_sd_write_sector()`
- Encapsulates SD card mounting, file validation, `.dsk` order probing

**`app_picker.c` / `app_picker.h`** (extract third — depends on app_sd):
- Owns the picker state machine (`s_sd_picker`)
- Exports `app_picker_open()`, `app_picker_handle_key()`, `app_picker_render()`, `app_picker_is_active()`
- Receives an `app_sd` context pointer for accessing the disk list

**Extraction order:** perf → SD → picker. Each extraction is a standalone commit. After all three, `app_main.c` should be under 1500 LOC, retaining: ROM loading, emulation loop, speed control, hotkey dispatch, audio integration.

**New files are placed in `main/`** alongside `app_main.c` (they are app-level, not reusable components).
