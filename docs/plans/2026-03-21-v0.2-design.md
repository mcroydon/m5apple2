# m5apple2 v0.2 Design

## Summary

Version 0.2 addresses correctness bugs in the Apple II bus emulation and 6502 CPU,
adds speaker audio output via I2S (ES8311 on Cardputer, NS4168 on ADV), and
implements disk write support for SD-card-mounted sector images (.dsk/.do/.po).

Work is organized into three checkpoints with build/test verification between each.

---

## Checkpoint 1: Bug Fixes & Correctness

### 1a. Keyboard Soft Switch Aliasing

**Problem:** Only `$C000` returns the key latch and only `$C010` clears the strobe.
Real hardware mirrors these across `$C000-$C00F` and `$C010-$C01F` respectively.

**Fix:** In `apple2_bus_read`, check address ranges before the switch statement:
- `$C000-$C00F`: return key latch
- `$C010-$C01F`: clear strobe, return latch value

In `apple2_bus_write`:
- `$C010-$C01F`: clear strobe
- `$C000-$C00F`: ignored (key latch is read-only)

**Files:** `components/apple2/apple2_machine.c`

### 1b. Game I/O Stubs

**Problem:** Reads from `$C061-$C067` fall through to main memory, causing games to
see random button/paddle state or hang in polling loops.

**Fix:** Add cases in `apple2_bus_read`:
- `$C061-$C063`: return `$00` (buttons not pressed, bit 7 clear)
- `$C064-$C067`: return `$00` (paddle timer expired, bit 7 clear)

**Files:** `components/apple2/apple2_machine.c`

### 1c. BCD SBC N/Z Flag Fix

**Problem:** On NMOS 6502, N and Z flags after SBC in decimal mode are set based on
the binary result before BCD adjustment. Current code sets them on the BCD-adjusted
result (which is 65C02 behavior).

**Fix:** In `cpu_sbc`, call `cpu_set_nz((uint8_t)diff)` before the decimal-mode
branch, and remove the `cpu_set_nz(cpu->a)` call after it.

**Files:** `components/apple2/cpu6502.c`

### 1d. Disk Nibble Multi-Advance

**Problem:** `apple2_disk2_tick` only advances one nibble per call even if the
accumulator indicates multiple bytes should have advanced. This can cause momentary
nibble stream stalls after track cache builds.

**Fix:** Change the `if (accum >= cpu_hz)` to a `while` loop, advancing nibble
position once per iteration. Cap at 8 iterations per tick to prevent runaway.

**Files:** `components/apple2/apple2_disk2.c`

### 1e. Test Verification

Run `apple2_core_tests`, `cardputer_keymap_tests`, and `apple2_rom_smoke` to verify
no regressions.

## Checkpoint 2: Speaker/Audio Output

### Architecture

New files in `components/cardputer/`:
- `cardputer_audio.c` / `cardputer_audio.h`

The audio subsystem:
- Initializes the I2S peripheral with codec-specific setup per variant
- ES8311 (Cardputer Original): requires I2C configuration of codec registers
- NS4168 (Cardputer ADV): I2S-only, no I2C config needed
- Maintains a circular sample buffer (512 samples at 22050 Hz, ~23ms latency)
- Exposes `cardputer_audio_init()` and `cardputer_audio_toggle(uint64_t cycle_count)`

### Integration

Replace `machine->speaker_toggles++` in `apple2_machine.c` with a callback mechanism.
The machine records toggle events with cycle timestamps. The main loop drains
accumulated toggles into the audio buffer after each CPU step slice. Between toggles
the output level is constant (+1/-1 alternating), producing an authentic 1-bit square
wave.

The I2S DMA pulls from the buffer asynchronously, so audio output does not block the
CPU loop. No separate FreeRTOS task is needed.

### Configuration

Add Kconfig options:
- Audio enable/disable
- Sample rate (default 22050 Hz)
- Variant-specific I2S pin assignments and codec selection

### Verification

Build and flash. Verify audio plays during boot beep and BASIC `PRINT CHR$(7)`.
Verify disk timing is unaffected by audio buffer fill overhead.

## Checkpoint 3: Disk Write Support & Timing

### Disk II Write Path

The Disk II write sequence on real hardware:
1. Software sets Q6=1, Q7=1 (write mode) via `$C0EF`
2. Software loads a byte into the data latch via `$C0ED`
3. The controller writes the latched byte at the current nibble position
4. The nibble position advances on the byte clock (same as reads)

Changes to `apple2_disk2.c`:
- When Q6=1, Q7=0 (`$C0ED`): latch the data bus value into `data_latch`
- When Q6=1, Q7=1: write mode. On each nibble tick, write `data_latch` into the
  track cache instead of reading from it
- Add `write_mode` flag derived from Q6 && Q7 state
- Add `track_cache_dirty` flag, set on any write to the track cache

### Dirty Track Flush

New function `apple2_disk2_flush_track()` that reverse-decodes GCR back to sectors:
1. Scan track cache for sector address field prologues (D5 AA 96)
2. Decode 4-and-4 header to extract volume/track/sector
3. Find data field prologue (D5 AA AD)
4. Reverse 6-and-2 GCR decoding to recover 256 bytes of sector data
5. Map physical sector to file sector using image order (DOS 3.3 or ProDOS)
6. Call `write_sector` callback to persist to SD

### SD Card Write-Back

New `app_sd_write_sector()` callback in `app_main.c` that seeks to the correct offset
in the image file and writes 256 bytes. Registered alongside the existing read_sector
callback via an additional parameter or new attach function.

### Flush Triggers

- Motor-off: natural save point; DOS 3.3 and ProDOS both turn motor off after writes
- Drive switch: prevents data loss when swapping active drives
- Disk eject/swap: via SD picker UI

Flushing is batched to motor-off rather than per-sector to avoid SD write latency
(~5-20ms per write) disrupting the emulation loop during multi-sector write sequences.

### Error Handling

If an SD write fails, log a warning. Data remains in the track cache and will be
retried on the next flush trigger. The emulator does not crash on write failure.

### Scope Exclusions

- Embedded (in-flash) disk images remain read-only
- `.nib` and `.woz` images remain read-only
- No write support for the `SECTOR_IMAGE` source kind (only `SECTOR_READER`)

### Track Cache Eager Rebuild

**Problem:** When the stepper seeks to a new track, the cache is lazily rebuilt on the
next `disk2_tick` call, causing a wall-clock stall during the CPU step.

**Fix:** In `disk2_set_phase`, after the quarter-track changes and cache is
invalidated, immediately call `disk2_build_track_cache`. This pays the cost during the
seek instruction rather than stalling on the first subsequent read. More realistic
(real hardware positions the head during the seek) and avoids timing discontinuities.

No FreeRTOS task or double-buffering needed. The cycle credit clamp in the main loop
absorbs the seek cost naturally.

### Verification

Run existing tests. Manually verify:
- Read path is unbroken (boot DOS 3.3, load software)
- Write path works (`SAVE HELLO`, `LOAD HELLO` round-trip)
- Seek timing feels consistent (no new stutter during track-to-track movement)
